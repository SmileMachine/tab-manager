# Design: 原生标签组管理器

## 背景
这个扩展面向长期打开大量标签页的 Edge 用户，同时保持 Chrome 兼容。主要使用场景不是快速点开一个小面板，而是在一个足够大的管理页面中搜索、筛选、选择、分组、跨窗口移动和关闭标签页。因此，独立管理页面是产品主体，popup 只作为进入管理页面的轻量入口。

扩展必须使用浏览器原生 Tab Group。它不维护另一套自定义分组关系，也不支持空 group，因为原生 tab group 不能脱离 tab 独立存在。

## 目标
- 管理所有浏览器窗口中的 tab 和原生 tab group。
- 在管理页面中保持浏览器原生 tab 顺序。
- 用轻量视觉方式呈现原生 group 归属，而不重新分类 tab。
- 支持高效率的搜索、筛选、选择和批量整理。
- 支持把选中的 tab 跨窗口移动到已有原生 group。
- 所有写操作完成后以浏览器真实状态为准刷新页面。
- 管理页面内的 group 展开/收起状态必须独立于浏览器原生 group collapsed 状态。

## 非目标
- 保存和恢复会话。
- 自动建议分组。
- 拖动整理。
- 域名聚合面板。
- 高级查询语法。
- 持久化搜索词、选择集、滚动位置或弹窗状态。

## 架构
扩展运行时由三个主要部分组成：

- 管理页面：React 应用，负责搜索、筛选、原生顺序渲染、选择模型、group 控件和批量操作。
- Popup：轻量入口，负责打开管理页面。
- Service worker：负责接收浏览器 tab、window、group 状态事件，并作为浏览器 API 访问边界。

浏览器 API 通过 `BrowserTabsApi` 适配层访问。应用逻辑不直接调用 `chrome.*`，而是依赖适配层方法。这样可以用 fake browser data 测试快照归并、命令规划和选择模型，而不要求测试运行在真实浏览器扩展环境中。

## 快照模型
管理页面读取所有窗口、tab 和原生 tab group，组成 `BrowserSnapshot`。模型保留浏览器原生标识：

- `windowId`
- `tabId`
- `groupId`
- `index`
- `pinned`、`active` 等 tab 状态
- group 的 `title`、`color` 等元数据

每个窗口视图按原生 `index` 排列 tab。Group 表示为有序 tab 列表中的一段区间。未分组 tab 保持在它的原生位置，不进入单独的容器。

## 主界面
管理页面按浏览器窗口分区展示。每个窗口内部严格按照浏览器原生 tab 顺序显示。属于同一 group 的连续 tab 区间使用非常淡的背景色，背景色来源于原生 group color。未分组 tab 使用默认中性背景。

Group 名称显示在左侧窄栏，并位于该 group 第一个 tab 的左侧。当用户滚动到这个 group 的范围内时，左侧 group 标签保持可见。该标签包含：

- checkbox
- group 名称
- tab 数量
- 展开/收起图标

Checkbox 用来选择或取消选择该 group 内所有 tab，并支持未选、半选、全选三种状态。展开/收起图标只改变管理页面里的视图状态。它不得调用 `tabGroups.update(..., { collapsed })`，也不得改变浏览器标签栏中的原生折叠状态。

Group 展开时显示内部 tab 行。Group 收起时压缩成一行摘要，摘要显示 group 名称、tab 数量、颜色标记和少量代表性域名。

## 搜索与筛选
第一版提供统一搜索，匹配 tab 标题、URL 和域名。同时提供低成本结构化筛选：

- 当前窗口、全部窗口、指定窗口。
- 全部 tab、已分组 tab、未分组 tab。
- 全部 tab、已固定 tab、未固定 tab。
- 指定 group。

搜索和筛选只影响管理页面的可见内容，不修改浏览器状态。搜索结果仍然保留窗口结构和原生 tab 顺序。

## 选择模型
选择状态以 `tabId` 存储。快照刷新后，仍然存在的 tab 继续保持选中；已经关闭或不可访问的 tab 从选择集中移除。如果已选 tab 跨窗口移动但 `tabId` 不变，它仍然保持选中。

选择状态是临时工作状态，不在重新打开管理页面时恢复。

## 命令
命令层把用户操作映射到浏览器原生操作：

- 用已选择的 tab 创建原生 group。
- 把已选择的 tab 移动到已有原生 group。
- 将已选择的 tab 移出 group。
- 重命名 group。
- 修改 group 颜色。
- 关闭单个 tab。
- 批量关闭已选择 tab。

创建 group 只在已选择 tab 全部属于同一窗口时启用。这符合原生 group 隶属于单个窗口的约束。移动到已有 group 允许跨窗口；此时选中的 tab 会移动到目标 group 所在窗口。

跨窗口移动时，选中的 tab 先按管理页面中的窗口显示顺序排列，再按各自源窗口中的原生 `index` 排列。命令应尽量把它们作为连续区间追加到目标 group 末尾。操作完成后，管理页面重新读取浏览器快照，并展示浏览器实际形成的顺序。

批量关闭必须确认。确认内容包括 tab 数量、涉及窗口数量、是否包含固定 tab，以及若干标题示例。跨窗口移动不确认，但操作栏必须在执行前显示目标 group 和目标窗口。

## 同步
管理页面打开时读取一次完整快照。之后监听 tab、window 和 group 变化事件。事件到达后经过短暂 debounce，再重新读取完整快照。第一版不做复杂的增量状态修正。

刷新后保留稳定 UI 状态：

- 搜索词。
- 当前筛选器。
- 窗口展开状态。
- 管理页面内的 group 展开状态。
- 仍然存在的已选 tab。

如果批量关闭确认框已经打开，而浏览器状态在确认前发生变化，则该确认失效，用户必须基于刷新后的对象集合重新确认。

## 偏好
只把稳定偏好保存到 `chrome.storage.local`：

- 默认窗口范围。
- 管理页面内的 group 展开/收起状态。
- 视图密度。

扩展不保存搜索词、选择集、滚动位置或打开中的弹窗。

## 错误处理
写操作完成后必须刷新快照。如果批量操作部分成功，界面应显示成功数量和失败数量，并以刷新后的浏览器真实状态为准。

如果浏览器在跨窗口移动时不能完全保持预期顺序，管理页面不得伪造本地顺序。它必须展示刷新后的真实浏览器顺序。

- 导出 tab 列表。

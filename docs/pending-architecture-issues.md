# 待解决问题：Manager 架构

本文档记录当前 `manager` 页面中已经暴露的结构性问题。这里列出的内容并不等同于立即实施范围；它们用于防止后续迭代只处理局部现象，而忽略长期影响维护成本的模块问题。

## 可选重构路径

### 方案 A：只做文件拆分

方案 A 只把 `App.tsx` 中的 JSX 组件和 helper 移到多个文件中，尽量不触碰状态组织、命令执行和拖拽模型。它的风险最低，能够快速降低单文件长度，但架构收益有限。如果拆分后的模块仍然通过大量 props 传递顶层状态，或者只是把原文件内容分散到多个浅模块，那么后续新增功能仍会遇到相同的理解成本。

### 方案 B：结构性模块拆分，并为后续架构重构预留位置

方案 B 是当前第一阶段建议执行的范围。它不仅移动文件，还明确模块责任：`components` 负责渲染和事件转发，`hooks` 负责生命周期和 React state 接入，`view` 负责纯派生数据。它仍然不改变用户可见行为，也不重写拖拽算法和命令执行模型。

这个方案的关键约束是：拆分不能把命令执行和拖拽计算进一步分散到展示组件中。也就是说，第一阶段虽然主要是低风险拆分，但拆分方向必须服务于后续 application 层和拖拽模型重构。

### 方案 C：重塑 manager 应用层和拖拽操作模型

方案 C 是更深的架构重构。它会引入明确的 manager application 模块，把用户动作、浏览器 API 调用、刷新、乐观更新、失败恢复和选择状态变化集中表达。同时，它会把拖拽真实执行计划、拖动预览和视觉状态统一到同一个操作模型中。

方案 C 的收益最高，但风险也最高。它不适合直接在当前 `App.tsx` 仍然混合组件、菜单、偏好、拖拽和命令执行的状态下开始。更合理的顺序是先完成方案 B，再为方案 C 单独编写 OpenSpec。

## 1. `App.tsx` 承担过多职责

`src/manager/App.tsx` 当前约 1700 行，同时负责页面装配、浏览器快照读取、偏好持久化、选择状态、拖拽状态、右键菜单、group 编辑框、批量关闭确认、浏览器命令执行、错误提示和大量 JSX 渲染。

这会造成两个直接后果。第一，理解一个用户动作需要在同一个文件内跨越多个概念区域。第二，新增功能时很容易继续把状态、回调和 DOM 细节放入顶层组件，使文件进一步膨胀。它已经不是一个单纯的 React 页面组件，而是页面组件、应用控制器和若干工具模块的混合体。

## 2. 拖拽预览和真实结果存在双重事实来源

拖拽相关逻辑分散在 `src/domain/commands.ts`、`src/domain/dragProjection.ts` 和 `WindowSection` 内部。

`planTabDrop()` 负责计算真实浏览器操作计划；`projectTabDropInView()` 负责拖动过程与乐观更新；`WindowSection` 再额外计算 projected rows、group label 高度、tab row 位置和拖动行颜色。之前多次出现“松手后的结果正确，但拖动过程显示不一致”的问题，说明真实操作和视觉投影并没有由同一个操作模型派生。

这是后续最值得进行架构重构的区域。最终应让一次拖拽计算同时产出浏览器操作计划、预览视图和显示状态，减少相似逻辑在多个模块中各自演化。

## 3. 命令规划和命令执行分离不完整

`src/domain/commands.ts` 已经包含若干纯函数，例如创建 group、移动到 group、释放内存、拖拽移动和批量关闭摘要的规划。这是合理的领域层基础。

问题在于命令执行仍然散布在 `App.tsx` 内部。执行函数不仅调用 `BrowserTabsApi`，还负责刷新快照、清除选择、打开编辑框、执行乐观更新、失败恢复和显示错误。这样一个用户动作横跨领域规划、浏览器 API、React state 和 UI 反馈，缺少稳定的应用动作层。

后续应引入 manager application 模块，把“用户意图如何变成浏览器写操作，以及写操作结束后如何更新页面状态”集中表达。

## 4. 弹层生命周期规则仍然分散

Escape stack 已经把 Escape 优先级集中处理，这是一个方向正确的改造。但右键菜单、group 编辑框和确认弹窗仍然分别处理外部点击、定位、滚动坐标、来源高亮和关闭行为。

这些规则本质上属于同一类 overlay 行为。继续分散会导致新增菜单或弹窗时复制相同逻辑，并重新引入多处判断。后续应形成统一的 overlay hooks，例如关闭规则、菜单定位和来源标记。

## 5. 视图派生流程不够显式

当前真实流程是：

1. 从浏览器读取 `BrowserSnapshot`。
2. 归并为 `BrowserSnapshotView`。
3. 应用搜索和结构化筛选。
4. 为每个窗口生成 `WindowRow`。
5. 根据拖拽状态生成预览视图和预览行。
6. 生成 group label、行位置、颜色和 drop 状态。
7. 交给 React 渲染。

其中前半部分已经有领域函数，后半部分仍然隐藏在 `WindowSection` 内部。结果是 JSX 组件必须理解许多派生规则，而这些规则其实更适合放入 presentation model。后续可以建立 `createManagerViewModel()` 或按窗口建立 `createWindowPresentation()`，让渲染层只消费已经整理好的行、标签和视觉状态。

## 6. 样式文件与 DOM 结构耦合过高

`src/manager/styles.css` 当前约 1000 行，覆盖页面框架、工具栏、列表、group rail、右键菜单、编辑框、dialog、拖拽状态和滚动条。它本身不一定需要立刻拆分，但它已经依赖当前 `App.tsx` 中的 DOM 结构。

因此样式整理应放在组件边界稳定之后进行。过早改动样式结构会把视觉回归风险与架构重构风险叠加。

## 7. 部分小工具函数重复或位置不稳定

例如 `domainFromUrl()` 在多个模块中出现，`groupsFromView()`、`tabIdsFromView()`、`updateGroupInView()`、`contextMenuPosition()` 等函数位于 `App.tsx` 底部，但它们并不是页面组件实现细节。

这些函数应按含义迁移到 domain、view 或 application 模块。迁移时不应只追求文件变小，而应让调用者获得更小、更稳定的接口。

## 8. 测试覆盖集中在领域纯函数，应用层行为较弱

当前测试覆盖了 snapshot、filters、selection、commands、dragProjection、windowRows、preferences 和 browserTabsApi。领域层基础较好。

应用层仍缺少更高层次的行为测试，例如创建 group 后清除选择、拖拽失败后恢复视图、菜单 action 对单个 tab 与已选 tab 的差异、overlay Escape 优先级等。第一阶段拆分不要求大量新增应用测试，但后续建立应用动作层时应补充对应测试。

## 建议处理顺序

第一阶段执行方案 B：进行结构性模块拆分，同时保留后续应用层和拖拽模型重构的位置。此阶段不改变用户可见行为。

第二阶段建立 manager application 模块，把命令执行、刷新、乐观更新、错误处理和选择状态变化集中表达。

第三阶段统一拖拽操作模型，让真实浏览器计划、预览视图和显示状态来自同一次计算。

第四阶段整理 overlay 行为和样式结构，在组件边界稳定后降低 UI 层复杂度。
